# Ludomaniac — Ludo Game Win Prediction

**Ludomaniac** is a machine learning project that simulates Ludo games to generate a synthetic dataset and train a model to predict winning combinations. The project covers the full data science pipeline: dataset generation, data cleaning, feature engineering, exploratory data analysis, model training, and deployment via a Streamlit dashboard.

# ![CI logo](https://codeinstitute.s3.amazonaws.com/fullstack/ci_logo_small.png)


## Dataset Content

The dataset is synthetically generated by simulating Ludo games between 4 players.

* **Size**: 10,000+ rows (minimum), generated across multiple simulated games
* **Data Types**: 4 numerical variables and 1 categorical variable
* **Variables**:
  - `Game_ID` — unique identifier for each simulated game
  - `Turn` — turn number within the game
  - `Player` — player colour: Red, Green, Yellow, or Blue
  - `Dice_Roll` — dice value rolled (1–6)
  - `Position` — player's board position after the move (0–52)
  - `Is_Winner` — 1 if the player won that game, 0 otherwise
* **Game Rules Simulated**:
  - 4 players start at position 0; first to reach position 52 wins
  - Overshooting position 52 causes a bounce-back
  - Rolling a 6 grants an extra turn
* **Source**: Synthetically generated via `jupyter_notebooks/create_dataset.ipynb`
* **Saved to**: `data file/Raw_Data/ludo_dataset.csv`


## Business Requirements

1. **Generate a Balanced Synthetic Dataset**: Simulate realistic Ludo games to produce a dataset with sufficient size (10,000+ rows) and balanced player win distribution for model training.
2. **Predict Winning Player**: Train a classification model that predicts which player will win a game based on dice rolls, board positions, and turn patterns.
3. **Identify Winning Patterns**: Analyse which in-game features (position reached, dice roll frequency, extra turns from 6s) most strongly correlate with winning outcomes.
4. **Deploy an Interactive Dashboard**: Build a Streamlit app allowing users to simulate a Ludo game and receive a win probability prediction in real time.


## Ethics, Roboethics, and Responsible Use

Although this project is framed around a game (and an educational business analogy), it still involves **automated recommendations** and **probability outputs**. That brings real AI/roboethics considerations.

- **Not a guarantee:** “Winner/Success Probability” is a model score learned from the project dataset (largely simulated). It is not a promise of outcomes, and it is not causal proof.
- **Automation bias:** Users may over-trust the top-ranked option, especially when two options are close. Treat near-ties as “roughly equivalent” and choose the option you can explain and justify.
- **Scope limits:** The Guided Play scenario generator follows simplified rules used for dataset generation. Real Ludo variants (safe squares, special tiles, house rules) can differ.
- **Representativeness / strategy bias:** Synthetic data can encode certain playstyles; recommendations may not generalize to all contexts.
- **Business analogy caution:** The “resource allocation” mode is for teaching OR/game theory/optimization patterns. Do not use it as a real operational decision engine without domain validation.
- **Privacy:** Avoid entering sensitive business information unless you control deployment, logging, and data-handling practices.

## Feature-Engineering Pipeline Rationale

| Pipeline Stage | Technique | Rationale |
|---|---|---|
| Correlation Analysis | Spearman + Pearson Rank Correlation | Identifies feature interactions with both linear and monotonic relationships for engineering target selection |
| Statistical Weighting | Mann-Whitney U Test | Ranks features by class-separation power (winner vs non-winner); weights engineered features by winner-predicting signal |
| Feature Creation | Transform Selection by Correlation Type | Strong-positive pairs use ratio transform; strong-negative use interaction; weak pairs use composite |
| Blended Weighting | Spearman × (0.25 + 0.75 × MWU_weight) | Preserves directional signal from correlation while amplifying features with strong discriminatory power |
| Post-Scaling | Secondary MWU Weighting | Reinforces class separation after nonlinear transforms; floor=0.10 prevents zeroing weak pairs |
| Model Training | Baseline → Tuned → Benchmarked | Establishes reference performance, optimizes class-weighted decision threshold, validates via alternative models |

## Analysis Techniques Used

### Feature-Engineering Methods:

1. **Correlation Analysis**:
   - Spearman Rank Correlation: Captures monotonic (non-linear) relationships between gameplay features
   - Pearson Correlation: Validates linear component of relationships
   - Pair Strength & Direction Classification: Strong/weak and positive/negative categorization for transform selection

2. **Statistical Significance Testing**:
   - Mann-Whitney U Test: Non-parametric test to measure class-separation power (winners vs non-winners)
   - Bounded Weight Conversion: U-statistics converted to normalized weights (0 to 1 range)
   - Significance Filtering: Only feature pairs with at least one statistically significant component included in engineering

3. **Feature Engineering Transforms**:
   - Ratio Transform: `f1 / (f2 + ε)` for strong-positive correlations
   - Interaction Transform: `f1 × f2` for strong-negative correlations
   - Composite Transform: `(f1 × f2) - (√|f1/f2| + f1/f2)` for weak-negative correlations

4. **Blended Statistical Weighting**:
`````
   - Correlation-MWU Blend: `c_blend = ρ × (0.25 + 0.75 × w_pair)`
   - Post-Scaling Reweighting: `f_eng_final = f_eng × max((w1+w2)/2, 0.10)`
   - Floor Protection: Minimum weight of 0.10 prevents zeroing out weaker feature pairs

### Model Development & Validation:

1. **Baseline Model**: Gradient Boosting classifier on engineered features for reference performance
2. **Constrained Hyperparameter Tuning**: Grid/random search with accuracy/precision floors to prevent degradation
3. **Calibration Testing**: Built-in + weighted Platt scaling to improve probability quality
4. **Cross-Validation**: Stratified 5-fold CV to validate tuned model robustness
5. **Alternative Model Benchmarking**: XGBoost, LightGBM, Random Forest, Neural Networks for exploratory validation
6. **Feature Importance Analysis**: Ranked feature contributions from tuned model to interpret decision-making

### Data-to-Model Structure Justification:

1. **Leakage-Aware Design**: Target (`Is_Winner`) and identifier (`Game_ID`) explicitly excluded from engineering to prevent information leakage
2. **Top-Down Validation**: Dataset → Correlation → Engineering → Baseline → Tuning → Benchmarking flow ensures each stage is validated before proceeding
3. **Statistical Rigor**: Blended Spearman+MWU weighting preserves both directional and discriminatory signals
4. **Threshold Optimization**: Production decision threshold selected via constrained search rather than default 0.5 assumption

## Ethical Considerations

### Model Fairness & Bias Issues:

1. **Balanced Game Simulation**: Dataset generated from fair Ludo rules applied equally to all 4 players
   - **Mitigation**: No inherent bias in rule application; all players have equal starting conditions and rule enforcement
   
2. **Data Representation**: Synthetic data may not capture all real-world Ludo playing styles or strategies
   - **Consideration**: Model predictions valid only within scope of simulated game dynamics
   - **Recommendation**: Validate against human player data before using for real-world applications
   
3. **Feature Importance & Interpretability**: Engineered features designed from statistical signals may not align with domain intuition
   - **Mitigation**: Step 7 includes feature importance analysis; visualizations help verify feature logic
   
4. **Model Deployment Scope**: Binary prediction (winner vs non-winner) assumes 4-player games; extends only to this configuration
   - **Recommendation**: Model retraining needed if game rules, player count, or board configuration change

### Responsible Prediction Use:

1. **No Determinism Claim**: Model provides probability estimates, not certainty; inherent randomness (dice rolls) limits predictability
   - **Communication**: Users informed that model captures patterns but cannot predict individual rolls
   
2. **Temporal Validity**: Model trained on synthetic data; performance may degrade over time if game dynamics evolve
   - **Recommendation**: Periodic retraining with updated game data to maintain accuracy
   
3. **Transparency**: All feature engineering decisions documented in notebook; formulas and weights publicly visible
   - **Benefit**: Enables external review and modification if desired

## Challenges & Resolutions

**Current Status**: Feature-engineering notebook complete and functional. All cells execute end-to-end successfully.

### Key Challenges Resolved:

1. **Feature-Engineering Weight Blending**:
   - **Challenge**: Balancing Spearman correlation signal vs Mann-Whitney U class-separation signal without over-weighting either
   - **Solution**: Implemented blended coefficient `c_blend = ρ × (0.25 + 0.75 × w_pair)` to give 25% baseline and 75% dynamic weighting
   - **Outcome**: Engineered features preserve correlation direction while amplifying discriminatory power
   - **Validation**: Post-scaling maintains non-zero feature values (floor=0.10) to prevent weak pairs from zeroing out

2. **Feature Leakage Prevention**:
   - **Challenge**: Need to use source features for correlation analysis while avoiding target leakage when engineering
   - **Solution**: Explicit exclusion of `Game_ID` and `Is_Winner` from engineering matrix; correlation computed on clean numeric subset only
   - **Outcome**: Safe feature engineering confirmed by reviewing dropped columns and protected columns lists

3. **Hyperparameter Tuning with Imbalanced Winners**:
   - **Challenge**: Class imbalance (most games won by only one of four players) makes recall optimization difficult
   - **Solution**: Constrained search with accuracy/precision floors; class-weight scaling; threshold optimization instead of default 0.5
   - **Outcome**: Tuned model achieves recall=0.738 (73.8% winner detection) while maintaining precision=0.660

4. **Probability Calibration for Decision Threshold**:
   - **Challenge**: Default threshold 0.5 sub-optimal for class-imbalanced data; tuned threshold may produce poorly calibrated probabilities
   - **Solution**: Tested both built-in and weighted Platt calibration; selected approach with best Brier score
   - **Outcome**: Calibrated probabilities enable threshold selection with confidence in predicted probabilities

### Knowledge Gaps Addressed:

1. **Statistically Informed Feature Engineering**:
   - **Gap**: Initially unfamiliar with blending correlation and hypothesis-test weights
   - **Resolution**: Researched Mann-Whitney U interpretation; understood how to normalize U-statistics to bounded weights
   - **Outcome**: Implemented principled weighting scheme combining two statistical signals

2. **Constrained Hyperparameter Optimization**:
   - **Gap**: Standard grid/random search doesn't enforce business constraints (min accuracy/precision floors)
   - **Resolution**: Implemented manual constraint-aware search with fallback scoring logic
   - **Outcome**: Tuned model meets specified performance floors while maximizing recall

3. **Cross-Validation for Model Stability**:
   - **Gap**: Needed to validate tuned settings beyond single train/test split
   - **Resolution**: Added stratified 5-fold CV with mean/std reporting for stability assessment
   - **Outcome**: Improved confidence in generalization to unseen data

### Implementation Notes (Technical)

This section explains the main implementation decisions in the Streamlit app.

#### Key goals

- Keep inference consistent with training-time feature engineering
- Make the dashboard robust when artifacts are missing
- Support multiple model artifacts without breaking the UI

#### Architecture

##### 1) Production artifact wrapper

The production model is stored as a joblib dict (not as a single sklearn estimator object).

The app wraps it into a small API:

- `ProductionLudoPredictor.predict_proba(X)`
- `ProductionLudoPredictor.predict(X)`

This keeps thresholding and calibration consistent.

##### 2) Feature engineering at inference

The app recreates engineered features at inference time so the dashboard can accept simple “raw” inputs.

Core steps:

- Convert numeric base columns to floats
- One-hot encode `Player` into a fixed set of columns
- Recreate engineered features and drop source features that were consumed
- Assemble the final model matrix in the exact `feature_columns` order expected by the artifact

Entry points:

- `_engineer_features(raw_df, artifacts)`
- `_prepare_model_matrix(raw_rows, predictor)`

##### 3) Cached artifacts

`_get_inference_artifacts()` builds and caches:

- Spearman correlation pairs (computed from the shipped dataset)
- Mann–Whitney U weight lookup table (embedded constants from the notebook)

This lets the app generate consistent engineered columns without re-running the notebook.

##### 4) Multi-model support

`load_model()` prefers a tuned RF artifact if present and falls back to the legacy GB artifact.

The Prediction page (Technical mode) also offers an experimental “train in-app” RF option. This is useful for demos but should not be treated as a production training pipeline.

##### 5) Diagnostics & export

The Diagnostics page provides:

- Cache clearing to avoid “stale wrapper” issues during hot reload
- Artifact existence checks
- Optional export helpers

#### Limitations / caveats

- If an artifact does not include full evaluation outputs, the app shows illustrative charts for confusion matrix / ROC curve / feature importance.

- Importing `streamlit_app.py` outside `streamlit run` is supported but will produce Streamlit “bare mode” warnings.
